Component Specification: Global File Preview Drawer
Component Name: FilePreviewDrawer
Role: A unified, global slide-out panel used to preview documents, spreadsheets, and artifacts across the entire application. It acts as a non-destructive overlay, allowing users to inspect files without losing their context on the main page.

1. Layout & Animation
Positioning: Fixed to the right side of the viewport (top-0 right-0 h-screen).

Animation: Slides in from the right (translateX(100%) to translateX(0)). Transition duration: ~300ms ease-in-out.

Backdrop (Overlay): A dark, semi-transparent overlay (bg-black/40 or bg-slate-900/50) covers the rest of the application.

Interaction: Clicking the backdrop closes the Drawer. (Since this is a "Preview" window, not a data-entry form, Overlay Click = Close is the correct UX pattern).

Z-Index: Must be higher than the main layout and sticky headers (e.g., z-50).

2. UI Architecture (Top to Bottom)
A. Top Header (System Bar)

Visual Style: Light gray background (bg-gray-50), compact height (approx. 40-48px), bottom border.

Left Side (Title): Contextual title indicating the source.

Example: "Attached files (10)" or "Generated Artifacts (3)".

Right Side (Controls):

Width Selector: A dropdown labeled "Window width" (Auto / 50% / 80% / 100%).

Close Button: A prominent [X] button (black background or high contrast icon) to dismiss the drawer.

B. Action Toolbar (File Context)

Visual Style: White background, sticky under the top header.

File Switcher (Dropdown): A wide select input displaying the current file's icon and name (e.g., 🔍 [XLS] Таблица 2. Порталы...).

UX Logic: If the user opened a file that belongs to a group (e.g., 10 attached files), they can switch between them directly from this dropdown without closing the Drawer.

Action Icons (Right-aligned):

🗑️ Delete (Red Trash): Deletes the file from the current list (triggers a confirmation popover if it affects an active operation).

☁️ Download (Blue Cloud): Triggers a direct download of the original file.

C. Main Content Viewer (The Canvas)

Visual Style: Takes up all remaining vertical space (flex-1, overflow-hidden).

Behavior: The rendering mechanism depends on the file extension:

XLSX / CSV: Renders a Data Grid component (like Handsontable or a customized shadcn Table) with locked headers.

PDF: Renders an embedded PDF viewer (iframe or react-pdf).

HTML/Web: Renders an isolated iframe.

3. Critical Logic & State Management

Rule 1: "Smart Width" Memory (localStorage) 

Logic: The drawer width is not strictly fixed. It respects user preference and remembers it across sessions.

Implementation:

When the user changes the width via the "Window width" dropdown (e.g., to 80%), save this value to browser localStorage under the key user_drawer_width.

The next time the Drawer opens (even for a different file), it must read localStorage and initialize at 80%.

Note: The "Auto" option should calculate width based on the file type (e.g., 80% for spreadsheets, 50% for text docs).

Rule 2: Global State Hoisting

The <FilePreviewDrawer /> component should ideally live at the highest possible layout level (e.g., inside AppLayout.tsx).

It is triggered by a global state or context (e.g., usePreviewStore(state => state.activeFile)). This ensures that clicking a file in the Left Panel (Brief) or the Right Panel (Report) opens the exact same Drawer instance without z-index conflicts.
