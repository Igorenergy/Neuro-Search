# Acuras Pro — Intelligent Deep Search Agent

## Overview

Acuras Pro is a B2B SaaS application for deep research and strategic intelligence. It uses a multi-agent AI architecture to conduct comprehensive investigations, extract structured data, and generate verified reports. The core UX philosophy is "Evidence Over Noise" and "No Dead Ends" — guiding users from vague queries to data-backed, verified reports.

The application has four main modules:
1. **Research Mode Dashboard** (`/research/dashboard`) — Mission control for monitoring active AI operations, system status, and research history
2. **Smart Search Launcher** (`/smart-search/new`) — Configuration wizard for launching new search tasks with contextual file uploads and deep-linked research
3. **Research Search** (`/research/search`) — Discovery interface for finding existing research projects with semantic/keyword search modes
4. **Assets Repository** (`/assets`) — Google Drive-like file manager for storing and organizing all artifacts (PDFs, CSVs, reports)

The visual style is "Cyber-Corporate" — a dark-themed, data-dense interface with electric blue primary accents and cyan highlights.

## System Architecture

### Frontend
- **Framework:** React 18 with TypeScript
- **Routing:** Wouter (lightweight client-side router)
- **State Management:** TanStack React Query for server state; React `useState` for local UI state
- **Styling:** Tailwind CSS v4 (via `@tailwindcss/vite` plugin) with a custom dark "Cyber-Corporate" color palette defined in `client/src/index.css` using CSS custom properties
- **Component Library:** shadcn/ui (new-york style) with Radix UI primitives — all UI components live in `client/src/components/ui/`
- **Build Tool:** Vite with React plugin, served from `client/` directory
- **Path Aliases:** `@/` → `client/src/`, `@shared/` → `shared/`, `@assets/` → `attached_assets/`

### Backend
- **Framework:** Express 5 running on Node.js with TypeScript (compiled via tsx)
- **HTTP Server:** Node `http.createServer` wrapping Express (supports potential WebSocket upgrades)
- **API Pattern:** RESTful JSON API under `/api/` prefix
- **Key Endpoints:**
  - `GET/POST /api/projects` — List and create research projects
  - `GET/PATCH /api/projects/:id` — Get and update a project
  - `GET/POST /api/projects/:id/plans` — Research plans per project
  - `GET/POST /api/projects/:id/files` — Files per project
  - `DELETE /api/files/:id` — Delete a file
- **Storage Layer:** `IStorage` interface in `server/storage.ts` with `DatabaseStorage` implementation using Drizzle ORM

### Database
- **Database:** PostgreSQL (required via `DATABASE_URL` environment variable)
- **ORM:** Drizzle ORM with `drizzle-zod` for schema validation
- **Schema Location:** `shared/schema.ts` (shared between client and server)
- **Tables:**
  - `users` — Basic user accounts (id, username, password)
  - `research_projects` — Research tasks with query, type, engine settings, cost estimates, status
  - `research_plans` — Versioned step-by-step plans linked to projects (steps stored as JSONB)
  - `research_files` — File metadata linked to projects
- **Migrations:** Drizzle Kit with `drizzle-kit push` for schema sync (`db:push` script)
- **IDs:** UUID v4 generated by PostgreSQL (`gen_random_uuid()`)

### Build & Development
- **Dev Mode:** `npm run dev` runs tsx on `server/index.ts`, which sets up Vite dev server as middleware with HMR
- **Production Build:** `npm run build` runs a custom build script (`script/build.ts`) that:
  1. Builds client with Vite (output to `dist/public/`)
  2. Bundles server with esbuild (output to `dist/index.cjs`), externalizing most deps but bundling a specific allowlist
- **Production Start:** `npm start` runs `node dist/index.cjs`
- **Static Serving:** In production, Express serves `dist/public/` with SPA fallback to `index.html`

### Key Design Patterns
- **Shared Schema:** Database schema and Zod validation types are defined once in `shared/schema.ts` and used by both frontend and backend
- **Storage Interface:** The `IStorage` interface abstracts database operations, making it possible to swap implementations
- **API Client Pattern:** `apiRequest` helper in `lib/queryClient.ts` handles fetch calls with error handling; React Query manages caching
- **SPA with API:** Client-side routing via Wouter with server-side API routes prefixed with `/api/`; all non-API routes fall through to `index.html`
